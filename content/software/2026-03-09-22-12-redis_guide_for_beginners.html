🔴 Redis 是什么？

Redis（Remote Dictionary Server）是一个开源的内存数据结构存储系统，它可以用作数据库、缓存和消息中间件。

💡 一句话理解

Redis 就像一个超级快速的"记事本"，把数据存在内存里，读写速度极快（每秒可处理 10万+ 次操作），适合存储临时数据、热点数据或需要快速访问的数据。

🎯 为什么选择 Redis？

  • ⚡ 极速性能：数据存在内存中，读写速度比传统数据库快 10-50 倍
  • 📊 丰富的数据类型：支持字符串、哈希、列表、集合、有序集合等
  • 💾 持久化：可以将内存数据保存到硬盘，防止数据丢失
  • 🔧 原子操作：所有操作都是原子性的，支持事务
  • 🌐 主从复制：支持数据备份和读写分离
  • 📈 高可用：支持哨兵和集群模式

📊 Redis 数据类型详解

Redis 不仅仅是简单的 key-value 存储，它支持多种丰富的数据类型，这是 Redis 强大的核心原因。

📝
String（字符串）
最基本的数据类型，可以存储字符串、数字或二进制数据
> set name "Alice"
> get name
"Alice"
📦
Hash（哈希）
键值对集合，适合存储对象（如用户信息）
> hset user:1 name "Bob" age 25
> hget user:1 name
"Bob"
📚
List（列表）
双向链表，支持从两端插入/弹出，适合做队列
> lpush queue task1
> rpop queue
"task1"
🎯
Set（集合）
无序不重复集合，支持交集、并集、差集运算
> sadd tags "redis" "cache"
> smembers tags
1) "redis" 2) "cache"
🏆
Sorted Set（有序集合）
带权重的集合，按分数排序，适合排行榜
> zadd ranking 100 "Alice" 80 "Bob"
> zrevrange ranking 0 -1
1) "Alice" 2) "Bob"
🎨
Bitmap / HyperLogLog
位图和基数统计，适合大规模数据统计
> setbit login:20240101 1001 1
> bitcount login:20240101
(integer) 1

🎯 实战应用场景

1️⃣ 缓存系统（最常用）

# 查询数据时先查 Redis，没有再查数据库并写入 Redis # 1. 尝试从 Redis 获取用户信息 GET user:1001 # 2. 如果不存在，从数据库查询后写入（设置 1 小时过期） SETEX user:1001 3600 '{"id":1001,"name":"Alice","age":25}' # 3. 删除缓存（用户数据更新时） DEL user:1001
📝 缓存更新策略
  • Cache Aside：先更新数据库，再删除缓存（最常用）
  • Read/Write Through：读写都经过缓存，缓存负责同步数据库
  • Write Behind：只写缓存，异步批量写入数据库

2️⃣ 会话存储（Session）

# 用户登录后，将 Session 存入 Redis # 存储 Session（30 分钟过期） SETEX session:abc123 1800 '{"user_id":1001,"username":"Alice","login_time":"2024-01-15"}' # 验证 Session GET session:abc123 # 延长 Session 时间（用户有操作时） EXPIRE session:abc123 1800 # 用户登出，删除 Session DEL session:abc123

3️⃣ 计数器（阅读量、点赞数）

# 文章阅读数 INCR article:1001:views # 获取当前阅读数 GET article:1001:views # 用户点赞（使用 Set 防止重复点赞） SADD article:1001:likes user:2001 # 获取点赞数 SCARD article:1001:likes # 检查用户是否已点赞 SISMEMBER article:1001:likes user:2001

4️⃣ 排行榜

# 游戏积分排行榜 # 添加玩家分数 ZADD game:ranking 5000 "player1" ZADD game:ranking 8000 "player2" ZADD game:ranking 6500 "player3" # 获取前 10 名（分数从高到低） ZREVRANGE game:ranking 0 9 WITHSCORES # 返回：1) "player2" 2) "8000" 3) "player3" 4) "6500" ... # 获取玩家排名 ZREVRANK game:ranking "player1" # 返回：(integer) 2（第 3 名，从 0 开始） # 增加玩家分数 ZINCRBY game:ranking 1000 "player1" # 获取玩家当前分数 ZSCORE game:ranking "player1"

5️⃣ 分布式锁

# 使用 SETNX（SET if Not eXists）实现简单的分布式锁 # 尝试获取锁（10 秒过期，防止死锁） SET lock:order:123 "owner_id" EX 10 NX # 返回 OK 表示获取成功，nil 表示已被占用 # 释放锁（先判断是不是自己加的锁，再删除） # Lua 脚本保证原子性 EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end" 1 lock:order:123 owner_id

6️⃣ 消息队列（简单版）

# 生产者：添加任务到队列 LPUSH task:queue '{"task":"send_email","to":"user@example.com"}' LPUSH task:queue '{"task":"generate_report","user_id":1001}' # 消费者：阻塞式获取任务（等待 30 秒） BRPOP task:queue 30 # 返回队列名称和任务内容，如果 30 秒内没有任务返回 nil # 发布/订阅模式（实时消息） # 订阅者执行： SUBSCRIBE channel:news # 发布者执行： PUBLISH channel:news "Breaking news!"

7️⃣ 限流器（Rate Limiter）

# 滑动窗口限流：限制用户每分钟最多 10 次请求 # 1. 记录当前请求（使用当前时间戳作为 score） ZADD rate:user:1001 [current_timestamp] [unique_request_id] # 2. 删除 60 秒前的记录 ZREMRANGEBYSCORE rate:user:1001 0 [current_timestamp - 60000] # 3. 统计当前窗口内的请求数 ZCARD rate:user:1001 # 4. 设置键的过期时间（60 秒） EXPIRE rate:user:1001 60 # 如果请求数 > 10，拒绝请求

💾 Redis 持久化

Redis 数据存在内存中，一旦服务器重启数据会丢失。Redis 提供两种持久化方式将数据保存到磁盘：

📸 RDB（Redis Database）

📋 什么是 RDB？

RDB 是快照方式，在指定时间间隔内将内存中的数据集快照写入磁盘。生成的 .rdb 文件是一个紧凑的二进制文件。

配置（redis.conf）

# 触发条件：900秒内至少1个key变化、300秒内至少10个key变化、60秒内至少10000个key变化 save 900 1 save 300 10 save 60 10000 # 禁用 RDB（注释掉所有 save 或设置） save "" # RDB 文件名 dbfilename dump.rdb # 保存目录 dir /var/lib/redis # 压缩 RDB 文件 rdbcompression yes # 校验 RDB 文件 rdbchecksum yes

手动触发保存

# 阻塞式保存（保存完成前不处理其他命令） SAVE # 后台保存（fork 子进程进行保存，主进程继续处理请求） BGSAVE # 查看最后一次保存时间 LASTSAVE

✅ RDB 优缺点

优点 缺点
文件紧凑，适合备份和恢复 可能丢失最后一次快照后的数据
恢复速度快 数据量大时 fork 子进程会耗时
对性能影响小（fork 子进程） 频繁快照影响性能

📝 AOF（Append Only File）

📋 什么是 AOF？

AOF 是日志方式，记录每个写操作命令，重启时重新执行这些命令恢复数据。AOF 文件是可读的文本格式。

配置（redis.conf）

# 开启 AOF appendonly yes # AOF 文件名 appendfilename "appendonly.aof" # 同步策略 # always：每次写入都同步（最安全，最慢） # everysec：每秒同步一次（推荐，平衡安全和性能） # no：由操作系统决定（最快，最不安全） appendfsync everysec # AOF 重写触发条件 # AOF 文件大小超过上次重写后的 100% 且至少 64MB 时触发 auto-aof-rewrite-percentage 100 auto-aof-rewrite-min-size 64mb

手动触发 AOF 重写

# 后台重写 AOF（压缩命令，去除冗余） BGREWRITEAOF
💡 AOF 重写是什么？

AOF 文件会越来越大，重写会创建一个新的 AOF 文件，只保留恢复当前数据所需的最小命令集。比如对一个 key 多次修改，只保留最后一次。

✅ AOF 优缺点

优点 缺点
数据安全性高（每秒同步） 文件体积通常比 RDB 大
只追加写入，不易损坏 恢复速度比 RDB 慢
AOF 文件可读，便于分析 性能开销比 RDB 大

🔄 混合持久化（Redis 4.0+）

同时开启 RDB 和 AOF，AOF 文件开头是 RDB 格式，后面是 AOF 格式。结合两者优点：

  • 恢复时先加载 RDB 部分（快），再执行 AOF 部分（完整）
  • 重写时生成新的混合文件
# 开启混合持久化（Redis 4.0+） aof-use-rdb-preamble yes

📊 持久化方案选择建议

场景 推荐方案
数据不重要，可丢失 关闭持久化
允许几分钟数据丢失 仅 RDB
只允许秒级数据丢失 仅 AOF（everysec）
不允许数据丢失 RDB + AOF（推荐）

⭐ 最佳实践与注意事项

🗝️ Key 命名规范

✅ 推荐做法
  • 使用冒号分隔：业务名:表名:ID，如 user:profile:1001
  • Key 长度适中（建议不超过 100 字节）
  • 避免使用特殊字符
  • 相同业务的 Key 前缀一致，方便管理

⚠️ 生产环境注意事项

🚨 危险命令
  • KEYS *：会遍历所有 Key，大数据量时会导致 Redis 卡住
  • FLUSHALL / FLUSHDB：清空所有/当前数据库数据
  • CONFIG SET：修改配置可能导致服务异常
  • DEBUG SEGFAULT：让 Redis 崩溃（调试用）

建议：通过 rename-command 配置禁用或重命名危险命令

# redis.conf - 禁用危险命令 rename-command FLUSHALL "" rename-command FLUSHDB "" rename-command KEYS ""

🔍 替代 KEYS 的方案

# 使用 SCAN 代替 KEYS（渐进式遍历，不会阻塞） SCAN 0 MATCH user:* COUNT 100 # 返回：下一个游标 + 匹配到的 key 列表 # 继续用返回的游标执行 SCAN 直到游标为 0

💡 性能优化建议

  • 使用 Pipeline：批量执行命令，减少网络往返
  • 控制 Value 大小：String 类型 Value 建议不超过 10KB，大 List/Hash 建议分片
  • 设置过期时间：为缓存数据设置合理的过期时间，避免内存无限增长
  • 使用连接池：避免频繁创建/关闭连接
  • 避免大 Key：定期使用 redis-cli --bigkeys 扫描大 Key
  • 合理使用数据类型：Hash 存储对象比 JSON String 更高效
